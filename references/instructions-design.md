# Instructions Design Guide

## 왜 지침이 중요한가

명확한 지침은:
- 모호성 감소
- 의사결정 향상
- 워크플로우 실행 순탄화
- 오류 감소

---

## 지침 작성 4단계

### Step 1: 기존 문서 활용

운영 절차, 지원 스크립트, 정책 문서를 LLM 친화적 루틴으로 변환.

**변환 전** (인간용 매뉴얼):
```
환불 요청 시 고객 정보를 확인하고,
주문 상태를 검토한 후 적절한 조치를 취합니다.
```

**변환 후** (LLM 지침):
```
환불 요청 처리:
1. get_customer_info(customer_id) → 고객 정보 확인
2. get_order_status(order_id) → 주문 상태 조회
3. IF 주문 상태 == "배송완료" AND 요청일 - 배송일 <= 14:
   → approve_refund(order_id) → 환불 승인
4. IF 조건 불충족:
   → 사유 안내 후 대안 제시
```

### Step 2: 작업 분해 (Task Decomposition)

밀도 높은 리소스를 "작고 명확한 단계"로 분해.

**원칙**:
- 각 단계는 하나의 명확한 행동
- 판단이 필요한 지점을 명시
- 분기 조건을 구체적으로

**분해 템플릿**:
```
[상황 설명]
1. [첫 번째 행동] - [사용할 도구]
2. [조건 확인] - IF [조건] THEN [행동A] ELSE [행동B]
3. [결과 처리] - [사용할 도구]
4. [완료 기준] - [검증 방법]
```

### Step 3: 명확한 액션 정의

모든 단계가 특정 액션 또는 출력에 대응.

**좋은 예**:
```
1. "주문번호를 알려주세요"라고 요청
2. get_order(order_id) 호출하여 주문 세부정보 검색
3. 주문 상태를 고객에게 안내
```

**나쁜 예**:
```
1. 고객의 주문을 확인합니다
2. 적절한 조치를 취합니다
```

### Step 4: 엣지 케이스 포착

**체크리스트**:
- [ ] 사용자가 불완전한 정보를 제공하면?
- [ ] 예상 외 질문이 들어오면?
- [ ] 도구 호출이 실패하면?
- [ ] 여러 해석이 가능한 요청이면?
- [ ] 범위 외 요청이 들어오면?
- [ ] 사용자가 화가 나 있으면?

**엣지 케이스 지침 예시**:
```
IF 사용자가 주문번호를 모르면:
  → "이메일 주소나 주문 시 사용한 이름을 알려주세요"
  → search_orders(email=...) 시도

IF 도구 호출 실패:
  → 최대 2회 재시도
  → 2회 실패 시 "잠시 후 다시 시도해 주세요" 안내
  → 에스컬레이션 고려
```

## 고급 기법

### 단일 유연한 기본 프롬프트
다수의 개별 프롬프트 대신 정책 변수를 받는 하나의 프롬프트.

```
당신은 {company_name}의 {role}입니다.
{company_policy}에 따라 고객을 지원합니다.
환불 한도: {refund_limit}
에스컬레이션 기준: {escalation_criteria}
```

**장점**: 유지보수/평가가 크게 단순화

### 도구 사용으로 결정론적 보장
"Tool-use can ensure that deterministic steps are deterministic and avoid hallucinations."

결정론적이어야 하는 단계 (계산, 데이터 조회 등)는 반드시 도구를 통해 처리.
에이전트가 추론으로 대체하지 않도록 명시적으로 지시.

## 지침 품질 체크리스트

- [ ] 모든 단계에 구체적 행동이 있는가?
- [ ] 판단 지점에 명확한 기준이 있는가?
- [ ] 분기 조건이 구체적인가?
- [ ] 엣지 케이스가 포함되었는가?
- [ ] 도구 사용 시점이 명시되었는가?
- [ ] 실패 시 행동이 정의되었는가?
- [ ] 에스컬레이션 기준이 있는가?
