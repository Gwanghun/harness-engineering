# Tool Design — ACI (Agent-Computer Interface) Principles

## 도구의 3가지 유형

### 데이터 도구 (Data)
정보를 수집하는 읽기 전용 도구.
- DB 쿼리, CRM 조회, 웹 검색, 문서 읽기, API GET

### 액션 도구 (Action)
외부 시스템을 변경하는 쓰기 도구.
- 이메일/문자 전송, 레코드 업데이트, 티켓 이관, 결제 처리

### 오케스트레이션 도구 (Orchestration)
다른 에이전트를 도구로 활용.
- 하위 에이전트 호출, 워크플로우 위임, 결과 수집

## ACI 설계 4원칙

### 1. 명확성 (Clarity)
"모델 입장에서 봤을 때 사용법이 명백한가?"

- 도구 이름은 동작을 명확히 설명: `get_order_details`, `send_email`
- 일관된 네이밍 프리픽스: `order_get`, `order_update`, `order_cancel`
- 파라미터 이름도 명확하게: `user_id` (O) vs `id` (X)

### 2. 문서화 (Documentation)
주니어 개발자용 docstring처럼 작성.

포함할 내용:
- 도구의 목적과 기능
- 사용 예시 (구체적 입력/출력)
- 엣지 케이스 처리 방법
- 입력 포맷 요구사항
- 다른 도구와의 경계 (이 도구를 쓸 때 vs 다른 도구를 쓸 때)
- 실패 시 행동

### 3. 포카요케 (Poka-yoke)
실수하기 어렵게 설계.

**적용 사례**:
- 상대 경로 → 절대 경로 필수 (SWE-bench 에이전트에서 검증됨)
- 필수 파라미터 명시, 선택 파라미터에 기본값 제공
- 유효하지 않은 입력 시 구체적 오류 메시지
- 위험한 작업에 확인 단계 추가

### 4. 자연스러운 포맷 (Natural Format)
모델이 학습한 형식에 가까울수록 우수.

**권장**:
- Markdown 코드 블록 (자연스러움)
- 충분한 사고 공간 제공 (모델이 생각할 토큰 여유)
- 포맷 오버헤드 제거 (수천 라인 수동 카운팅 불필요)

**비권장**:
- 과도한 JSON 이스케이프 요구
- 복잡한 문자열 인코딩

## 도구 명세 템플릿

각 도구를 설계할 때 다음 항목을 정의:

```
도구 이름: [action_verb]_[resource]
설명: 한 문장으로 도구의 목적
유형: 데이터 | 액션 | 오케스트레이션
위험 등급: LOW | MEDIUM | HIGH

입력 파라미터:
  - name: 파라미터 이름
    type: 타입
    required: true/false
    description: 설명
    example: 예시 값

출력:
  - 성공 시 반환 형식
  - 실패 시 오류 메시지 형식

사용 시기: 이 도구를 선택해야 하는 조건
사용하지 말 것: 다른 도구를 써야 하는 조건
엣지 케이스: 알려진 특이 상황과 처리 방법
```

## 도구 문서화의 중요성

"Documenting tools thoroughly makes them discoverable to agents."

에이전트가 올바른 도구를 선택하려면:
- 각 도구의 용도를 명확히 이해해야 함
- 사용 금지 사항도 알아야 함
- 도구 간 경계가 분명해야 함

**도구 사용의 가치**: "Tool-use can ensure that deterministic steps are deterministic and avoid hallucinations."
→ 결정론적이어야 하는 단계는 도구로 처리하여 환각 방지

## 도구 수 가이드라인

- **15개 미만**: 단일 에이전트로 충분
- **15+ 명확히 구분된 도구**: 여전히 단일 에이전트로 가능
- **10개 미만이지만 중복되는 도구**: 에이전트가 혼동할 수 있음 → 분리 고려
- **도구 과부하 징후**: 에이전트가 잘못된 도구를 자주 선택
