# Orchestration Patterns Reference

## Pattern Decision Tree

```
기능 요구사항 분석
├─ 단일 단계 작업인가? → 단순 LLM 호출 (에이전트 불필요)
├─ 고정된 순차 단계인가? → Prompt Chaining
├─ 입력 유형별 다른 처리가 필요한가? → Routing
├─ 독립적 병렬 처리가 가능한가? → Parallelization
├─ 동적 작업 분해가 필요한가? → Orchestrator-Workers
├─ 반복 품질 개선이 필요한가? → Evaluator-Optimizer
├─ 중앙에서 여러 전문 에이전트를 조율하는가? → Manager Pattern
└─ 에이전트 간 순차적 제어 이전이 필요한가? → Handoff Pattern
```

## 1. Prompt Chaining

각 LLM 호출이 이전 출력을 처리하는 순차 파이프라인.

**구조**: `입력 → LLM 1 → [게이트 검사] → LLM 2 → [검증] → 출력`

**적합 상황**:
- 작업이 고정된 부분 과제로 명확히 분해 가능
- 정확도 향상을 위해 지연시간을 거래할 수 있음
- 각 단계 사이에 검증 게이트가 필요

**예시**: 마케팅 카피 생성 → 번역 → 검수

**장점**: 각 LLM 호출이 단순해져 성능 향상
**단점**: 지연시간 증가, 체인이 길어지면 오류 누적

---

## 2. Routing

입력을 분류한 뒤 전문화된 경로로 분배.

**구조**: `입력 → 분류기 → [경로 A | 경로 B | 경로 C]`

**적합 상황**:
- 서로 다른 카테고리가 별도 처리 필요
- 분류가 정확하게 수행 가능
- 비용 최적화가 중요 (간단한 질문 → 경량 모델)

**예시**: 고객 서비스에서 일반 질문/환불/기술지원으로 라우팅

**장점**: 관심사 분리, 비용 최적화
**단점**: 분류 오류가 전체에 영향

---

## 3. Parallelization

독립적인 부분 과제를 동시에 실행.

### Sectioning (섹셔닝)
작업을 독립적 부분으로 분할하여 병렬 실행.
- 가드레일: 쿼리 처리 LLM + 부적절 콘텐츠 필터링 LLM 분리

### Voting (투표)
동일 작업을 여러 번 실행하여 다양한 출력 수집 후 합산.
- 코드 보안 검토: 여러 프롬프트가 독립적으로 취약점 검사

**장점**: 속도 향상 또는 신뢰도 증가
**단점**: 비용 증가

---

## 4. Orchestrator-Workers

중앙 LLM이 동적으로 작업을 분해하고 워커에 위임 후 결과 통합.

**핵심 차이점**: 병렬화와 달리 부분 과제가 사전에 정해지지 않음.

**적합 상황**:
- 필요 부분 과제를 미리 예측 불가
- 입력에 따라 작업 범위가 달라짐

**예시**: 코딩 에이전트 — 변경 필요 파일 수와 각 파일의 변경 성질은 작업 의존적

**장점**: 복잡한 문제 해결의 유연성
**단점**: 오케스트레이션 오류 위험

---

## 5. Evaluator-Optimizer

한 LLM이 응답 생성, 다른 LLM이 평가하고 피드백 제공 (반복).

**구조**: `입력 → 생성자 ⟷ 평가자 (피드백 루프) → 만족 시 출력`

**적합 상황**:
- 명확한 평가 기준 존재
- 반복 개선이 측정 가능한 가치 제공

**예시**: 문학 번역 — 평가자가 뉘앙스 부족 지적 후 개선

**장점**: 점진적 품질 향상
**단점**: 반복으로 인한 비용/지연시간 증가

---

## 6. Manager Pattern (OpenAI)

중앙 에이전트가 전문화된 에이전트 네트워크를 도구로 보유하며 조율.

**적합 상황**:
- 여러 전문 영역을 하나의 워크플로우에서 조율
- 결과를 중앙에서 수집하고 통합

**예시**: 번역 매니저 → 프랑스어/이탈리아어/스페인어 전담 에이전트

---

## 7. Handoff Pattern (OpenAI)

에이전트 간 순차적 제어 이전. 각 에이전트가 사용자와 직접 상호작용.

**적합 상황**:
- 트리아지 후 전문 에이전트로 라우팅
- 중앙 제어 불필요

**예시**: 웰컴 에이전트 → 불만 처리/환불/기술지원 에이전트로 이관

---

## 패턴 비교 매트릭스

| 패턴 | 복잡도 | 지연시간 | 유연성 | 비용 | 적합 규모 |
|------|--------|---------|--------|------|-----------|
| Prompt Chaining | 낮음 | 중간 | 낮음 | 낮음 | 소-중 |
| Routing | 낮음 | 낮음 | 중간 | 최적화 | 중 |
| Parallelization | 중간 | 낮음 | 중간 | 높음 | 중 |
| Orchestrator-Workers | 높음 | 높음 | 높음 | 높음 | 대 |
| Evaluator-Optimizer | 중간 | 높음 | 중간 | 높음 | 중 |
| Manager | 높음 | 중간 | 높음 | 중간 | 대 |
| Handoff | 중간 | 낮음 | 높음 | 중간 | 중-대 |

## 핵심 원칙

1. **단순하게 시작**: 단일 에이전트의 역량을 먼저 최대화
2. **필요할 때만 확장**: 더 간단한 솔루션이 충분하지 않을 때만 복잡성 추가
3. **단순하고 초점 맞춘 에이전트가 낫다**: 관심사 분리로 인지적 부하 감소
