# Evaluation System Design

## 평가의 목적

에이전트가 의도한 대로 동작하는지 검증하고, 개선 방향을 제시하기 위한 체계.

---

## 평가 설계 프로세스

### Step 1: 성공 기준 정의

사용자와 함께 정의할 핵심 메트릭:

**기능 메트릭**:
- 작업 완료율 (Task Completion Rate)
- 정확도 (Accuracy)
- 도구 선택 정확도 (Tool Selection Accuracy)
- 최종 결과 품질 (Output Quality)

**성능 메트릭**:
- 응답 지연시간 (Latency)
- 토큰 사용량 (Token Usage)
- API 호출 횟수 (API Calls)
- 비용 (Cost per Task)

**안전 메트릭**:
- 가드레일 위반율 (Guardrail Violation Rate)
- 에스컬레이션 비율 (Escalation Rate)
- 오류 복구율 (Error Recovery Rate)

### Step 2: 베이스라인 수립

1. **최강 모델로 시작**: 가장 능력 있는 모델로 기준선 설정
2. **평가 데이터셋 구축**: 10-50개의 대표적 테스트 케이스
3. **기준 점수 기록**: 각 메트릭의 현재 수준 측정

### Step 3: 최적화

1. **모델 축소 시도**: 성능 목표를 유지하면서 작은 모델로 교체
2. **프롬프트 최적화**: 지침을 조정하여 성능 개선
3. **도구 설계 개선**: 도구 명세를 조정하여 선택 정확도 향상
4. **비용-정확도 트레이드오프**: 수용 가능한 수준에서 비용/지연 최적화

## 테스트 케이스 설계

### 케이스 유형

**Happy Path**: 정상적인 사용 시나리오
- 전형적인 사용자 요청
- 예상되는 도구 호출 순서
- 정상적인 결과

**Edge Cases**: 경계 조건
- 불완전한 정보 제공
- 모호한 요청
- 여러 해석이 가능한 입력

**Adversarial Cases**: 적대적 시나리오
- 프롬프트 인젝션 시도
- 범위 외 요청
- 가드레일 우회 시도

**Failure Cases**: 실패 시나리오
- 도구 호출 실패
- 타임아웃
- 잘못된 입력 데이터

### 케이스 작성 템플릿

```
케이스 ID: TC-001
카테고리: Happy Path | Edge | Adversarial | Failure
설명: 테스트 목적

입력: 사용자 요청 내용
기대 행동:
  1. 기대되는 도구 호출
  2. 기대되는 중간 단계
  3. 기대되는 최종 출력

성공 기준:
  - 작업 완료 여부
  - 정확도 임계값
  - 가드레일 준수 여부

메트릭 기대값:
  - 최대 도구 호출 횟수
  - 최대 응답 시간
```

---

## 평가 파이프라인

```
테스트 데이터셋
  ↓
자동 실행 (에이전트에 테스트 케이스 투입)
  ↓
결과 수집 (실제 행동 기록)
  ↓
자동 비교 (기대 vs 실제)
  ↓
메트릭 계산 (완료율, 정확도, 비용 등)
  ↓
리포트 생성
  ↓
개선 사항 도출
```

---

## 반복 개선 사이클

1. **평가 실행** → 현재 수준 측정
2. **병목 식별** → 실패 케이스 분석
3. **개선 적용** → 지침/도구/가드레일 수정
4. **재평가** → 개선 효과 검증
5. **반복** → 목표 달성까지

## 핵심 원칙

- 평가 없이 배포하지 않기
- 실패 사례에서 가장 많이 배움
- 메트릭은 비즈니스 목표와 연결되어야 함
- 평가 데이터셋은 지속적으로 확장
