# Verification & Testing Guide

생성된 하네스의 품질을 검증하기 위한 방법론.

---

## 1. 구조 검증

### 파일 존재 확인

모든 에이전트 정의 파일과 스킬 파일이 올바른 위치에 있는지 확인한다.

**체크리스트**:
- [ ] `.claude/agents/{name}.md` — 모든 에이전트 파일 존재
- [ ] `.claude/skills/{name}/skill.md` — 모든 스킬 파일 존재
- [ ] 오케스트레이터 스킬 파일 존재

### Frontmatter 검증

각 스킬의 YAML frontmatter가 올바른지 확인:

```yaml
---
name: skill-name        # 필수
description: "..."      # 필수, 적극적 트리거 포함
---
```

### 참조 일관성 확인

- 에이전트 정의에서 참조하는 스킬이 실제 존재하는지
- 스킬에서 참조하는 레퍼런스 파일이 실제 존재하는지
- 오케스트레이터에서 참조하는 에이전트가 모두 정의되어 있는지

### 데이터 흐름 검증

- 각 단계의 출력 경로가 다음 단계의 입력 경로와 일치하는지
- `_workspace/` 파일명 컨벤션이 일관되게 사용되는지
- 최종 산출물 경로가 명확히 지정되어 있는지

---

## 2. 트리거 검증

스킬의 description이 의도대로 작동하는지 테스트한다.

### Should-Trigger 쿼리 작성 (8~10개)

스킬을 트리거해야 하는 다양한 표현을 작성한다:

**다양성 기준**:
- 공식적 표현 vs 캐주얼 표현
- 명시적 요청 vs 암시적 요청
- 한국어 vs 영어
- 전문가 표현 vs 초보자 표현

**예시 (harness-engineering 기준)**:
```
1. "AI 에이전트 하네스를 설계해줘"
2. "에이전트 가드레일이랑 오케스트레이션 패턴 잡아줘"
3. "agent harness design"
4. "챗봇 만들건데 도구 설계부터 평가까지 전체 설계 필요"
5. "LLM 기반 자동화 시스템의 안전장치를 구성하고 싶어"
6. "에이전트 팀 구성해서 QA까지 포함한 아키텍처 설계"
7. "human-in-the-loop 전략 수립"
8. "에이전트 기반 기능 만들기"
```

### Should-NOT-Trigger 쿼리 작성 (8~10개)

키워드가 유사하지만 이 스킬이 아닌 다른 도구/스킬이 적합한 "near-miss" 쿼리:

**작성 핵심**: 명백히 무관한 쿼리는 테스트 가치가 없다. 경계가 모호한 쿼리가 좋은 테스트 케이스.

**예시**:
```
1. "MCP 서버 만들어줘" → mcp-builder 스킬
2. "이 코드의 테스트 케이스 작성해줘" → tdd 스킬
3. "하네스 마켓플레이스 플러그인 설치" → 설치 관련
4. "기존 에이전트 정의 파일 수정해줘" → 직접 편집
5. "코드 리뷰해줘" → code-review 스킬
6. "프롬프트 엔지니어링 팁 알려줘" → 일반 질문
7. "Claude API 사용법" → claude-api 스킬
8. "배포 자동화 파이프라인 구축" → devops 영역
```

### 충돌 확인

기존 스킬과 트리거가 겹치는 부분이 있는지 확인:
- 같은 키워드에 여러 스킬이 반응하면 description을 조정하여 경계를 명확히 함

---

## 3. 실행 테스트

### 테스트 프롬프트 작성

각 스킬에 대해 2~3개의 현실적인 테스트 프롬프트를 작성한다. 실제 사용자가 입력할 법한 구체적이고 자연스러운 문장으로 작성.

**좋은 테스트 프롬프트**:
- 구체적인 도메인 컨텍스트 포함
- 실제 프로젝트에서 나올 법한 요구사항
- 모호성이 있어 에이전트의 판단력을 테스트

**예시**:
```
프롬프트 1: "고객 문의를 자동 분류하고 답변하는 에이전트를 만들려고 해. 
            환불, 배송, 기술지원 3가지 카테고리로 나눠야 하고, 
            환불은 5만원 이상이면 사람 승인이 필요해."

프롬프트 2: "코드 리뷰 에이전트 팀을 구성하고 싶어. 
            보안 검사, 성능 분석, 코드 스타일 각각 전담 에이전트가 필요해."
```

### With-Skill vs Without-Skill 비교

가능하면 스킬 있는 실행과 없는 실행을 비교하여 부가가치를 확인한다:
- **With-skill**: 스킬을 읽고 작업 수행
- **Without-skill (baseline)**: 같은 프롬프트를 스킬 없이 수행

### 결과 평가

**객관적 검증 가능 시** (파일 생성, 데이터 추출 등):
- assertion 형태로 통과 기준을 정의
- 예: "에이전트 정의 파일이 생성되었는가", "가드레일 섹션이 포함되었는가"

**주관적 평가 시** (문체, 설계 품질 등):
- 사용자 피드백에 의존

### 반복 개선 루프

테스트에서 문제 발견 시:
1. 피드백을 **일반화**하여 스킬 수정 (특정 예시에만 맞는 좁은 수정 금지)
2. 수정 후 재테스트
3. 의미 있는 개선이 더 이상 없을 때까지 반복

## 4. 드라이런 테스트

실제 실행 없이 전체 파이프라인의 논리적 흐름을 검증한다.

### 단계 순서 검증

9단계 파이프라인이 논리적으로 올바른지 확인:
- 각 단계의 전제 조건이 이전 단계에서 충족되는가?
- 건너뛸 수 있는 단계와 필수 단계가 명확히 구분되는가?

### I/O 매칭 검증

각 단계의 출력이 다음 단계의 입력으로 정확히 연결되는지 확인:

```
Stage 1 산출물 (적합성 결과) → Stage 2 입력 (요구사항)
Stage 2 산출물 (패턴 + 모드) → Stage 3 입력 (에이전트 구조)
Stage 3 산출물 (에이전트 파일) → Stage 4 입력 (에이전트별 지침)
Stage 4 산출물 (스킬 파일) → Stage 5 입력 (도구 요구사항)
...
```

### 폴백 경로 검증

- 에러 시나리오별 대응 경로가 정의되어 있는지
- 폴백 경로가 실제로 도달 가능한지
- 무한 루프나 데드록 가능성이 없는지

### 실행 모드별 검증

- **에이전트 팀 모드**: 팀원 간 통신 경로, 작업 의존성, 팀 크기 적정성
- **서브 에이전트 모드**: 각 에이전트의 입출력 연결, run_in_background 설정

---

## 5. 테스트 시나리오 문서화

오케스트레이터 스킬에 `## 테스트 시나리오` 섹션을 추가한다.

### 정상 흐름 시나리오 (최소 1개)

```
시나리오: 고객 서비스 에이전트 하네스 설계
1. Stage 1: 적합성 진단 → 3개 기준 중 2개 해당 (복잡한 의사결정 + 비정형 데이터)
2. Stage 2: Routing 패턴 선택, 에이전트 팀 모드
3. Stage 3: 분류기, 환불 전문가, 배송 전문가, QA 에이전트 정의
4. Stage 4: 각 에이전트별 스킬 파일 생성
5. ...
기대 결과: 4개 에이전트 파일 + 4개 스킬 파일 + 오케스트레이터 + 검증 통과
```

### 에러 흐름 시나리오 (최소 1개)

```
시나리오: Stage 5에서 도구 설계 실패
1. Stage 5에서 외부 API 스펙이 불완전
2. 에러 감지 → 사용자에게 API 스펙 보완 요청
3. 보완 후 Stage 5 재실행
4. 나머지 단계 정상 진행
기대 결과: 에러가 전파되지 않고 해당 단계에서 해결
```
