# Human-in-the-Loop Design

## 인간 개입이 필요한 2가지 트리거

### 1. 실패 임계값 초과

에이전트가 스스로 해결하지 못하는 상황.

**설계 항목**:
- 재시도 제한: 최대 N회 시도 후 사람에게 이관
- 의도 파악 실패: 사용자 의도를 이해하지 못할 때
- 루프 감지: 동일 작업을 반복할 때
- 최대 반복 횟수: 에이전트 실행 전체의 상한선

**구현 패턴**:
```
시도 횟수 추적
  IF 시도 >= max_retries:
    → 현재 상태와 시도 이력을 포함하여 사람에게 에스컬레이션
    → 에이전트는 대기 상태로 전환
```

### 2. 고위험 행동

영향이 크거나 되돌릴 수 없는 행동.

**고위험 행동 예시**:
- 주문 취소
- 대규모 환불 승인
- 결제 처리
- 계정 삭제
- 대량 데이터 수정
- 외부 서비스 연동 (이메일 발송 등)

**설계 원칙**: 에이전트 신뢰도가 충분히 향상될 때까지 사람 감독 필수

## 체크포인트 설계 패턴

### Approval Gate
특정 행동 전에 사람의 승인을 요구.

```
에이전트가 행동 계획 수립
  → 계획을 사람에게 제시
  → 승인 대기
  → 승인 시 실행, 거부 시 재계획
```

### Review Checkpoint
결정적 단계에서 사람이 결과를 검토.

```
에이전트가 중간 결과 생성
  → 사람에게 검토 요청
  → 피드백 수렴
  → 피드백 반영 후 다음 단계 진행
```

### Confidence-Based Escalation
에이전트의 확신도에 따라 동적으로 결정.

```
IF confidence >= 0.9: 자동 실행
IF 0.7 <= confidence < 0.9: 사람에게 알림 (자동 실행)
IF confidence < 0.7: 사람의 승인 필요
```

## 점진적 자율성 확대 전략

에이전트의 자율성을 단계적으로 높이는 접근.

### Phase 1: 관찰 모드
- 에이전트가 행동을 제안만 함
- 모든 행동은 사람이 실행
- 에이전트의 판단 정확도 측정

### Phase 2: 저위험 자동화
- LOW 등급 행동만 자동 실행
- MEDIUM/HIGH는 여전히 사람 승인
- 자동 실행 결과를 사후 검토

### Phase 3: 중위험 자동화
- LOW + MEDIUM 등급 자동 실행
- HIGH만 사람 승인
- 이상 탐지 시 자동 중단

### Phase 4: 감독된 자율성
- 대부분 자동 실행
- 이상 행동 탐지 시 자동 에스컬레이션
- 정기적 사후 감사

## 배포 전략

### 반복적 접근 (전부 아니면 전무 금지)

```
1. 단일 에이전트로 시작 (최소 기능)
2. 실제 사용자와 소규모 검증
3. 피드백 기반 개선
4. 점진적 역량 확대
5. 실패 식별 및 엣지 케이스 발견
6. 평가 사이클 반복
```

### 배포 체크리스트

- [ ] 최대 반복 횟수가 설정되었는가?
- [ ] 에스컬레이션 경로가 정의되었는가?
- [ ] 고위험 행동에 승인 게이트가 있는가?
- [ ] 실패 시 사람에게 전달되는 컨텍스트가 충분한가?
- [ ] 롤백 계획이 있는가?
- [ ] 모니터링/로깅이 설정되었는가?
- [ ] 초기 자율성 수준이 적절한가?
