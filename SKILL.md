# Harness Engineering Workflow

AI 에이전트 기능의 하네스(가드레일, 오케스트레이션, 도구 설계, 평가, 인간 개입)를 대화형으로 설계하고 실행 가능한 산출물을 생성. OpenAI와 Anthropic의 에이전트 구축 가이드에 기반.

## Initial Offer

사용자에게 워크플로우를 소개:

> AI 에이전트 기능의 하네스를 설계해 보겠습니다. 9단계로 진행합니다:
> 1. **적합성 진단** — 에이전트가 맞는지 판단
> 2. **오케스트레이션 설계** — 실행 모드와 패턴 선택
> 3. **에이전트 정의** — 에이전트 파일 생성
> 4. **지침 및 스킬 설계** — LLM 지침 + 스킬 파일 작성
> 5. **도구 설계** — ACI 원칙 기반 도구 명세
> 6. **가드레일** — 계층적 안전장치 구성
> 7. **평가 시스템** — 메트릭과 테스트 케이스
> 8. **인간 개입 + 배포** — 에스컬레이션과 점진적 출시
> 9. **검증 및 테스트** — 구조/트리거/드라이런 검증
>
> 어떤 기능을 만드시나요?

거부 시 자유 형식. 수락 시 Stage 1부터.

## Stage 1: 적합성 진단

**목표**: 에이전트가 정말 필요한지 판단하고, 사용자 수준에 맞게 진행 깊이를 조절.

3가지 기준으로 질문:
1. **복잡한 의사결정** — 미묘한 판단, 예외, 맥락 민감성 필요?
2. **규칙 유지보수 난이도** — 방대한 규칙, 높은 업데이트 비용?
3. **비정형 데이터 의존** — 자연어 해석, 문서 분석 필요?

**판정**: 0개 해당 → 단순 LLM 호출/워크플로우 안내. 1개+ → Stage 2로.

추가로 프로젝트 유형(웹 앱/API/봇), 기술 스택, 기존 코드베이스 여부 파악.

**사용자 숙련도 감지**: 대화의 맥락 단서(사용 용어, 질문 수준)로 기술 수준을 파악하고 이후 진행 깊이를 조절한다.
- **초급**: 각 단계마다 개념 설명 + 예시 제공. "assertion", "JSON schema" 같은 전문 용어에 부연 설명 추가.
- **중급**: 핵심 질문 위주로 진행. 레퍼런스 필요시 로드.
- **고급**: 단계 병합/생략 허용. 바로 산출물 생성 가능.

**산출물**: 적합성 진단 결과 요약

## Stage 2: 오케스트레이션 설계

**목표**: 실행 모드, 최적 패턴, 데이터 전달 방식을 결정.

### 2-1. 실행 모드 선택

에이전트가 2개 이상 필요한 경우, 실행 모드를 먼저 결정한다:

| 기준 | 에이전트 팀 | 서브 에이전트 |
|------|------------|-------------|
| 에이전트 간 통신 | SendMessage로 직접 소통 | 메인에게만 결과 반환 |
| 조율 방식 | 자체 조율 (TaskCreate) | 오케스트레이터가 직접 호출 |
| 적합 상황 | 발견 공유, 상충 토론, 누락 보완 필요 | 독립적 결과 전달만 필요 |

**기본값은 에이전트 팀**. 에이전트가 1개뿐이거나 통신이 불필요하면 서브 에이전트를 선택.

### 2-2. 패턴 선택

**참조**: [orchestration-patterns.md](references/orchestration-patterns.md) — 7가지 패턴의 결정 트리, 비교 매트릭스
**참조** (라우팅 구현이 필요할 때): [claw-code-patterns.md §1 Token Scoring Router](references/claw-code-patterns.md) — 토큰 스코어링 기반 경량 라우터 의사코드

요구사항 분석 후 추천:
- Prompt Chaining / Routing / Parallelization / Orchestrator-Workers / Evaluator-Optimizer
- Manager Pattern / Handoff Pattern

Routing 패턴을 선택한 경우, 구현 방식을 결정:
- **토큰 스코어링 라우터**: LLM 없이 결정론적 매칭 (비용 0, 지연 0). 도구/커맨드 수 50개 이하에 적합.
- **LLM 기반 분류기**: 의미적 이해 필요 시. 비용/지연 있으나 유연.
- **하이브리드 (Two-Level Dispatch)**: 토큰 스코어링으로 후보 선별 → LLM이 최종 선택. 대규모 인벤토리에 권장.

**원칙**: "단일 에이전트로 시작, 필요할 때만 복잡성 추가"

### 2-3. 데이터 전달 프로토콜

에이전트 간 데이터 흐름을 정의한다:

| 전략 | 방식 | 적합한 경우 |
|------|------|-----------|
| **메시지 기반** | SendMessage로 직접 통신 | 실시간 피드백, 가벼운 상태 전달 |
| **태스크 기반** | TaskCreate/TaskUpdate로 상태 공유 | 의존 관계 관리, 진행 추적 |
| **파일 기반** | 약속된 경로에 파일 읽기/쓰기 | 대용량 데이터, 감사 추적 필요 |

파일 기반 시 `_workspace/` 폴더에 중간 산출물 저장. 파일명 컨벤션: `{phase}_{agent}_{artifact}.{ext}`

추천 패턴 + 근거 제시 → 사용자 확인. 구현 계획이 필요하면 `/plan` 스킬 연동.

**산출물**: 실행 모드, 선택된 패턴, 데이터 전달 방식, 아키텍처 다이어그램

## Stage 3: 에이전트 정의 생성

**목표**: 실행 가능한 에이전트 정의 파일을 생성.

**모든 에이전트는 `.claude/agents/{name}.md` 파일로 정의한다.** 에이전트 정의 파일 없이 Agent 도구의 prompt에 역할을 직접 넣지 않는다. 파일로 존재해야 세션 간 재사용이 가능하고, 팀 통신 프로토콜이 명시되어야 협업 품질이 보장되기 때문이다.

각 에이전트 파일의 필수 섹션:
1. **핵심 역할** — 이 에이전트가 무엇을 하는가
2. **작업 원칙** — 판단 기준과 우선순위
3. **입력/출력 프로토콜** — 받는 데이터와 산출물 형식
4. **에러 핸들링** — 실패 시 행동
5. **팀 통신 프로토콜** (팀 모드 시) — 메시지 수신/발신 대상과 작업 요청 범위

**모델 설정**: Agent 도구 호출 시 `model: "opus"` 파라미터를 명시한다. 하네스의 품질은 에이전트의 추론 능력에 직결된다.

**QA 에이전트 포함 시**: `general-purpose` 타입을 사용한다 (`Explore`는 읽기 전용이므로 검증 스크립트 실행 불가). QA의 핵심은 "존재 확인"이 아니라 **"경계면 교차 비교"** — API 응답과 프론트 훅을 동시에 읽고 shape을 비교하는 것이다.

**산출물**: `.claude/agents/{name}.md` 파일들

---

## Stage 4: 지침 및 스킬 설계

**목표**: LLM 친화적 지침을 작성하고, 재사용 가능한 스킬 파일을 생성.

### 4-1. 지침 설계

**참조**: [instructions-design.md](references/instructions-design.md) — 4단계 작성법, 품질 체크리스트

질문:
1. 기존 운영 절차/문서 존재 여부
2. 주요 작업 흐름
3. 알려진 엣지 케이스

작업: 기존 문서 → 단계별 지침 변환, 도구 사용 시점 명시, 분기 조건과 엣지 케이스 포함.

### 4-2. 스킬 파일 생성

각 에이전트가 사용할 스킬을 `.claude/skills/{name}/skill.md`에 생성한다.

**스킬 구조**:
```
skill-name/
├── skill.md       (필수 — YAML frontmatter + Markdown 본문)
├── scripts/       (반복/결정적 작업용 실행 코드)
├── references/    (조건부 로딩하는 참조 문서)
└── assets/        (출력에 사용되는 파일)
```

**작성 원칙**:
| 원칙 | 설명 |
|------|------|
| **Why를 설명하라** | "ALWAYS/NEVER" 대신 이유를 전달. LLM은 이유를 이해하면 엣지 케이스에서도 올바르게 판단. |
| **Lean하게 유지** | skill.md 본문 500줄 이내. 초과 시 references/로 분리. |
| **일반화하라** | 특정 예시에만 맞는 좁은 규칙보다 원리를 설명. 오버피팅 금지. |
| **Description은 적극적으로** | 스킬이 하는 일 + 구체적 트리거 상황을 모두 기술. |

**Progressive Disclosure (3단계 로딩)**:

| 단계 | 로딩 시점 | 크기 목표 |
|------|----------|----------|
| Metadata (name + description) | 항상 컨텍스트에 존재 | ~100단어 |
| skill.md 본문 | 스킬 트리거 시 | <500줄 |
| references/ | 필요할 때만 | 무제한 |

**산출물**: 시스템 프롬프트 + `.claude/skills/{name}/skill.md` 파일들

## Stage 5: 도구(Tool) 설계

**목표**: ACI 원칙으로 도구 명세.

**참조**: [tool-design-aci.md](references/tool-design-aci.md) — 도구 유형, 4원칙, 명세 템플릿
**참조** (도구 등록/실행 구조 설계 시): [claw-code-patterns.md §2 Execution Registry](references/claw-code-patterns.md) — Spec→Registry→Execution 3계층 패턴

순서:
1. 필요 도구 식별 (데이터/액션/오케스트레이션)
2. 명세 작성 (이름, 파라미터, 입출력, 엣지 케이스)
3. ACI 적용: 명확성, 문서화, 포카요케, 자연스러운 포맷
4. 도구 간 경계 정의
5. 위험 등급 분류 (LOW/MEDIUM/HIGH)
6. **도구 등록 구조 결정**: 도구 수 10개 이상이거나, 환경별 도구 세트가 다르거나, 권한 기반 필터링이 필요하면 Execution Registry 패턴 적용 검토

MCP 서버 구현 시 `mcp-builder` 스킬 연동.

**산출물**: 도구 명세서 + 코드 스캐폴딩

---

## Stage 6: 가드레일 설계

**목표**: 계층적 안전장치 구성.

**참조**: [guardrails-catalog.md](references/guardrails-catalog.md) — 6가지 유형, 낙관적 실행, 체크리스트
**참조** (도구 권한 제어 구현 시): [claw-code-patterns.md §3 Two-Stage Permission](references/claw-code-patterns.md) — 정적 필터 + 동적 승인 2단계 모델

체크리스트를 사용자와 검토:
- 프라이버시/평판 위험? PII? 외부 쓰기? 재정 영향? 비가역 행동?

해당 위험에 맞는 가드레일 선택. 낙관적 실행 패턴 적용 (병렬 실행, 위반 시 개입). HIGH 등급은 사전 승인.

**도구 권한 제어가 필요한 경우**, Two-Stage Permission 모델 적용 검토:
- **Stage 1 (정적 필터)**: 환경별 deny list로 위험 도구 사전 차단 (비용 0)
- **Stage 2 (동적 승인)**: Allow/Deny/Prompt 3모드 정책 + Prompter 추상화로 CLI/Web/CI 환경 대응

**산출물**: 가드레일 설계서 + 구현 코드

---

## Stage 7: 평가(Eval) 시스템 설계

**목표**: 품질 측정과 개선 체계.

**참조**: [evaluation-system.md](references/evaluation-system.md) — 메트릭, 테스트 케이스 템플릿

진행:
1. 성공 기준 정의 (기능/성능/안전 메트릭)
2. 테스트 케이스 설계 (Happy/Edge/Adversarial/Failure)
3. 베이스라인 수립 계획
4. 최적화 방향 (모델 축소, 프롬프트/도구 개선)

`tdd` 스킬로 테스트 코드 구현 가능.

**산출물**: 평가 메트릭 + 테스트 케이스 목록

---

## Stage 8: 인간 개입 + 배포 전략

**목표**: 에스컬레이션과 점진적 배포.

**참조**: [human-in-the-loop.md](references/human-in-the-loop.md) — 체크포인트, 자율성 단계

설계:
1. **실패 임계값**: 재시도 제한, 최대 반복 횟수
2. **고위험 행동**: Stage 5의 HIGH 등급 도구 기반
3. **체크포인트**: Approval Gate / Review / Confidence-Based
4. **자율성 수준**: 관찰 → 저위험 → 중위험 → 감독된 자율성
5. **모니터링/로깅**

**산출물**: 인간 개입 설계서 + 배포 로드맵

---

## Stage 9: 검증 및 테스트

**목표**: 생성된 하네스가 실제로 동작하는지 검증.

**참조**: [verification-testing.md](references/verification-testing.md) — 검증 방법론 상세

### 9-1. 구조 검증
- 모든 에이전트 파일이 `.claude/agents/`에 존재하는지 확인
- 스킬의 frontmatter (name, description) 검증
- 에이전트 간 참조 일관성 확인
- 파일 간 데이터 흐름에 빈 구간(dead link)이 없는지 확인

### 9-2. 트리거 검증
각 스킬의 description이 의도대로 트리거되는지 검증:
- **should-trigger 쿼리 8~10개** — 다양한 표현 (공식적/캐주얼, 명시적/암시적)
- **should-NOT-trigger 쿼리 8~10개** — 키워드가 유사하지만 다른 스킬이 적합한 "near-miss" 쿼리
- 기존 스킬과의 트리거 충돌도 확인

### 9-3. 실행 테스트
- 테스트 프롬프트 2~3개 작성 (현실적인 사용자 입력)
- with-skill vs without-skill 비교 실행으로 부가가치 확인
- 객관적 검증 가능 시 assertion 기반 평가, 주관적이면 사용자 피드백
- 문제 발견 시 피드백을 일반화하여 스킬 수정 후 재테스트

### 9-4. 드라이런
- 전체 파이프라인의 단계 순서 논리 검증
- 각 단계의 출력이 다음 단계의 입력과 매칭되는지 확인
- 에러 시나리오별 폴백 경로 실행 가능성 확인
- 에이전트 팀 모드 시 통신 경로, 작업 의존성, 팀 크기 적정성 확인

## Tips

- **톤**: 직접적, 절차적. 근거를 간략히 설명하되 판매하지 않음
- **편의**: Stage 건너뛰기 허용. 건너뛴 이유 기록
- **컨텍스트**: 이전 Stage 결정이 다음에 영향. 항상 참조
- **Progressive Disclosure**: 각 단계의 레퍼런스는 해당 단계 진입 시에만 로드
- **스킬 연동**: Stage 2 → `/plan`, Stage 5 → `mcp-builder`, Stage 7 → `tdd`, 완료 → `/plan-eng-review`
